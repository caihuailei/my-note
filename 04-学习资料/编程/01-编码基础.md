ASCII编码的主要缺点包括：

1. **字符集有限** - 只能表示128个字符（标准ASCII）或256个字符（扩展ASCII），无法表示全球大多数语言的字符

2. **不支持多语言** - 无法表示中文、日文、韩文、阿拉伯文、俄文等非英文字符

3. **没有emoji支持** - 无法表示表情符号等现代常用字符

4. **扩展不统一** - 扩展ASCII（128-255）有多种不同的编码标准（如ISO-8859-1、Windows-1252等），不同系统间可能不兼容

5. **国际化能力差** - 在全球化时代，ASCII的局限性非常明显，无法满足多语言环境的需求

这些缺点推动了Unicode编码（如UTF-8、UTF-16）的发展，Unicode可以表示全球所有主要语言的字符。

## Unicode（万国码/万国码）

**Unicode**（统一码、万国码）是一种行业标准，旨在为全球所有语言的字符分配唯一的数字编码，解决不同国家和地区之间字符编码不兼容的问题。

### 核心概念

| 概念 | 说明 |
|------|------|
| **UCS (Universal Character Set)** | Unicode标准定义的字符集，包含全球所有主要字符 |
| **码点 (Code Point)** | Unicode中每个字符的唯一数字编号，格式为U+XXXX |
| **编码形式 | 将码点转换为字节序列的实际编码方式 |

### 常见编码形式

| 编码 | 特点 | 用途 |
|------|------|------|
| **UTF-8** | 可变长度（1-4字节），向后兼容ASCII | Web、Linux、现代系统主流 |
| **UTF-16** | 可变长度（2或4字节），中文常用2字节 | Windows、Java、JavaScript |
| **UTF-32** | 固定长度（4字节） | 内存处理简单，但占用空间大 |

### Unicode的特点

1. **包含字符数量巨大** - 目前包含超过14万个字符（Unicode 15.1），涵盖：
   - 拉丁字母、西里尔字母、阿拉伯字母等
   - 中文（CJK）、日文（Japanese）、韩文（Hangul）
   - 古文字、数学符号、emoji等

2. **持续更新** - Unicode联盟定期发布新版本，添加新字符和emoji

3. **代码区域划分**：
   - U+0000-U+007F: ASCII
   - U+4E00-U+9FFF: 中日韩统一表意文字
   - U+AC00-U+D7AF: 韩文音节
   - U+1F600-U+1F64F: 表情符号

4. **向后兼容** - UTF-8完全兼容ASCII（前128个字符编码相同）

现代信息系统普遍采用UTF-8作为默认编码，它已成为互联网和跨平台开发的行业标准。

## UTF-8

**UTF-8**（8-bit Unicode Transformation Format）是Unicode的一种可变长度编码格式，是目前互联网和现代系统中最主流的字符编码方式。

### 工作原理

| 字节数 | 码点范围 | 二进制格式（x表示数据位） |
|--------|----------|---------------------------|
| 1字节 | U+0000 - U+007F | `0xxxxxxx` |
| 2字节 | U+0080 - U+07FF | `110xxxxx 10xxxxxx` |
| 3字节 | U+0800 - U+FFFF | `1110xxxx 10xxxxxx 10xxxxxx` |
| 4字节 | U+10000 - U+10FFFF | `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx` |

### 特点

| 特性 | 说明 |
|------|------|
| **向后兼容ASCII** | ASCII字符（0-127）用单字节表示，且值不变 |
| **自同步** | 可通过字节首部的`1`和`0`判断字符边界 |
| **效率高** | 英文文本与ASCII一样节省空间；中文等常用3字节 |
| **无字节序问题** | 字节序固定，无需BOM（Byte Order Mark） |

### 示例

| 字符 | Unicode码点 | UTF-8编码（十六进制） |
|------|-------------|----------------------|
| A | U+0041 | `41` |
| 中 | U+4E2D | `E4 B8 AD` |
| 😀 | U+1F600 | `F0 9F 98 80` |

### 应用现状

- **Web标准** - 超过98%的网站使用UTF-8
- **现代操作系统** - Linux、macOS默认UTF-8
- **编程语言** - Python 3、JavaScript、Java默认UTF-8
- **数据库** - MySQL、PostgreSQL推荐UTF-8

## 为什么会乱码？

**乱码**（Mojibake）产生的根本原因是：**文本的编码方式与解码方式不匹配**。

### 常见原因

1. **编码/解码不一致** - 最常见
   ```
   场景：用UTF-8编码保存，却用GBK解码
   UTF-8编码：E4 BD A0 E5 A5 BD
   按GBK解码：ÎÒÃ的身体 → 乱码
   ```

2. **扩展ASCII兼容性问题** - 中文在不同编码中字节数不同

3. **缺少编码声明** - HTML缺少`<meta charset="UTF-8">`

4. **数据库编码不一致** - 存储和读取使用不同编码

### 解决思路

**核心原则：编码和解码必须使用同一种字符集**

```
正确：原文 → [UTF-8编码] → 存储 → [UTF-8解码] → 正确显示
错误：原文 → [UTF-8编码] → 存储 → [GBK解码] → 乱码
```

### 预防措施

1. **统一使用UTF-8** - 所有环节采用相同编码
2. **明确声明编码** - 在文件头、HTTP头、数据库连接中声明
3. **避免手动转换** - 使用编程语言内置的编码处理函数
4. **检测编码** - 无法确定时，用工具检测文件编码
