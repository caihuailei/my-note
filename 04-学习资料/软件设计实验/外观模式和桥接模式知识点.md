# 外观模式（Facade）与桥接模式（Bridge）

> 两者都是**结构型设计模式**，但解决的问题和核心思想完全不同：
> - **外观模式**：简化复杂系统的调用 → "把复杂的事变简单"
> - **桥接模式**：分离抽象与实现 → "把耦合的事拆独立"

---

## 一、外观模式（Facade Pattern）

### 1. 核心定义

> 为复杂子系统提供一个统一、简化的高层接口，隐藏子系统内部的复杂交互。

- **核心思想**：简化调用，隐藏复杂
- **本质**：封装复杂性，提供便捷入口

---

### 2. 结构组成

```
┌─────────────────────────────────┐
│        Client (客户端)          │
│      只需调用 Facade            │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│     Facade (外观类)             │
│  统一对外接口，协调各子系统     │
└──────┬────────┬────────┬───────┘
       │        │        │
       ▼        ▼        ▼
  SubSystem  SubSystem  SubSystem
  (子系统1)   (子系统2)   (子系统3)
```

| 角色 | 说明 |
|------|------|
| **Facade** | 外观类，统一对外接口，内部组合/调用多个子系统类 |
| **SubSystem** | 子系统类，实现具体功能的独立类 |
| **Client** | 客户端，仅依赖外观类 |

---

### 3. 代码示例（Java）-home theater

```java
// ========== 子系统类 ==========

// 子系统1：灯光
class Light {
    public void on()  { System.out.println("灯光打开"); }
    public void off() { System.out.println("灯光关闭"); }
}

// 子系统2：空调
class AirConditioner {
    public void cool() { System.out.println("空调制冷"); }
    public void off()  { System.out.println("空调关闭"); }
}

// 子系统3：窗帘
class Curtain {
    public void close() { System.out.println("窗帘关闭"); }
    public void open()  { System.out.println("窗帘打开"); }
}

// ========== 外观类 ==========

class HomeTheaterFacade {
    private Light light;
    private AirConditioner ac;
    private Curtain curtain;

    public HomeTheaterFacade() {
        light = new Light();
        ac = new AirConditioner();
        curtain = new Curtain();
    }

    // 统一接口：观影模式
    public void watchMovie() {
        light.off();
        ac.cool();
        curtain.close();
        System.out.println("🎬 开始观影");
    }

    // 统一接口：结束观影
    public void endMovie() {
        light.on();
        ac.off();
        curtain.open();
        System.out.println("👋 观影结束");
    }
}

// ========== 客户端 ==========

public class Client {
    public static void main(String[] args) {
        HomeTheaterFacade facade = new HomeTheaterFacade();
        facade.watchMovie();  // 只需调用一个方法
        facade.endMovie();
    }
}
```

**运行结果：**
```
灯光关闭
空调制冷
窗帘关闭
🎬 开始观影
灯光打开
空调关闭
窗帘打开
👋 观影结束
```

---

### 4. 适用场景

| 场景 | 说明 |
|------|------|
| 📦 复杂子系统 | 子系统复杂、依赖多，客户端直接调用易出错 |
| 🎫 第三方SDK | 为复杂系统提供简单入口（如支付SDK、云服务SDK） |
| 🔗 解耦需求 | 降低客户端与子系统的耦合度 |

**反例**（不使用外观模式）：
```java
// 客户端需要知道所有子系统细节
light.off();
ac.cool();
curtain.close();
fan.stop();
...
// 客户端代码冗长且易出错！
```

---

### 5. 优缺点

| 优点 | 缺点 |
|------|------|
| ✅ 简化客户端调用 | ⚠️ 可能违反开闭原则 |
| ✅ 解耦客户端与子系统 | ⚠️ 无法 bypass facade 时灵活性不足 |
| ✅ 提高子系统独立性 | |

## 二、桥接模式（Bridge Pattern）

### 1. 核心定义

> 将抽象部分与实现部分分离，使两者可以独立变化。

- **核心思想**：分离抽象与实现，组合替代继承
- **解决痛点**：避免多维度变化导致的**类爆炸**

---

### 2. 结构组成

```
┌─────────────────────────────────────────────┐
│          Abstraction (抽象层)                │
│          - 定义高层接口                      │
│          - 持有 Implementor 引用  ← 桥接!   │
└──────────────┬──────────────────────────────┘
               │
     ┌─────────┴─────────┐
     │                   │
     ▼                   ▼
┌──────────────┐  ┌──────────────────┐
│ RefinedAb    │  │  RefinedAb       │
│ 具体抽象1     │  │  具体抽象2        │
└──────────────┘  └──────────────────┘

┌─────────────────────────────────────────────┐
│       Implementor (实现层接口)               │
└──────────────┬──────────────────────────────┘
               │
     ┌─────────┴─────────┐
     │                   │
     ▼                   ▼
┌──────────────┐  ┌──────────────────┐
│ ConcreteImp  │  │  ConcreteImp     │
│ 具体实现1     │  │  具体实现2        │
└──────────────┘  └──────────────────┘
```

| 角色 | 说明 |
|------|------|
| **Abstraction** | 抽象层，定义高层接口，持有实现层引用 |
| **RefinedAbstraction** | 扩展抽象，具体子类 |
| **Implementor** | 实现层接口，定义底层实现 |
| **ConcreteImplementor** | 具体实现，实现层接口的具体类 |

---

### 3. 代码示例（Java）- 绘图系统

**场景**：绘制不同**形状**（圆形、方形）和不同**颜色**（红、蓝）的图形

```java
// ========== 实现层：颜色接口 ==========

interface Color {
    void applyColor();
}

class RedColor implements Color {
    public void applyColor() { System.out.print("填充红色 "); }
}

class BlueColor implements Color {
    public void applyColor() { System.out.print("填充蓝色 "); }
}

// ========== 抽象层：形状类 ==========

abstract class Shape {
    protected Color color;  // 桥接！组合实现层

    public Shape(Color color) {
        this.color = color;
    }

    abstract void draw();
}

class Circle extends Shape {
    public Circle(Color color) { super(color); }
    public void draw() {
        System.out.print("绘制圆形，");
        color.applyColor();
        System.out.println();
    }
}

class Square extends Shape {
    public Square(Color color) { super(color); }
    public void draw() {
        System.out.print("绘制方形，");
        color.applyColor();
        System.out.println();
    }
}

// ========== 客户端 ==========

public class Client {
    public static void main(String[] args) {
        // 组合：红色圆形 + 蓝色方形
        Shape redCircle = new Circle(new RedColor());
        Shape blueSquare = new Square(new BlueColor());

        redCircle.draw();
        blueSquare.draw();

        // 甚至可以动态切换颜色！
        Shape greenCircle = new Circle(new Color() {
            public void applyColor() { System.out.print("填充绿色"); }
        });
        greenCircle.draw();
    }
}
```

**运行结果：**
```
绘制圆形，填充红色 
绘制方形，填充蓝色 
绘制圆形，填充绿色 
```

---

### 4. 类爆炸对比

**不使用桥接模式**（继承方式）：

| 类名 | 说明 |
|------|------|
| `RedCircle` | 红色圆形 |
| `BlueCircle` | 蓝色圆形 |
| `RedSquare` | 红色方形 |
| `BlueSquare` | 蓝色方形 |
| `RedTriangle` | 红色三角形 |
| `BlueTriangle` | 蓝色三角形 |
| ... | ... |

✅ 2种颜色 × 3种形状 = **6个类**

❌ 如果增加 **绿色**，需要新增 3 个类 → **9个类**

❌ 如果增加 **三角形**，需要新增 2 个类 → **8个类**

---

**使用桥接模式**（组合方式）：

| 层级 | 类 |
|------|---|
| **实现层** | `RedColor`, `BlueColor`, `GreenColor` |
| **抽象层** | `Circle`, `Square`, `Triangle` |

✅ 3种颜色 + 3种形状 = **6个类**（固定）

---

### 5. 适用场景

| 场景 | 说明 |
|------|------|
| 🌐 多维度变化 | 设备品牌 + 功能、支付方式 + 渠道 |
| 📦 独立扩展 | 新增维度不影响现有代码 |
| 🔄 动态切换 | 运行时改变实现 |

---

### 6. 优缺点

| 优点 | 缺点 |
|------|------|
| ✅ 避免类爆炸 | ⚠️ 增加系统复杂度（需理解双层结构） |
| ✅ 符合开闭原则 | ⚠️ 不适合维度稳定的情况 |
| ✅ 提高扩展性 | |

## 三、核心区别对比

| 维度 | 外观模式 (Facade) | 桥接模式 (Bridge) |
|------|-------------------|-------------------|
| **核心目的** | 简化复杂系统的调用 | 分离抽象与实现 |
| **解决痛点** | 调用太麻烦 | 类太多（类爆炸） |
| **结构方式** | 组合多个子系统 | 组合抽象与实现 |
| **变化维度** | 通常单维度（系统复杂度） | 多维度（正交变化） |
| **客户端视角** | "简化壳" | "双层结构" |
| **典型场景** | 前端调用后端 SDK | 形状+颜色、品牌+功能 |

---

## 四、一句话总结

| 模式 | 口诀 |
|------|------|
| **外观模式** | "把复杂的事变简单" → 给复杂系统套一层"简化壳" |
| **桥接模式** | "把耦合的事拆独立" → 分离抽象与实现，用组合替代继承 |

---

## 五、快速判断题

**Q1**: 客户端需要调用很多子系统，但只需要一个简单功能？
→ 使用 **外观模式**

**Q2**: 有两个独立变化的维度，子类数量会指数级增长？
→ 使用 **桥接模式**

**Q3**: 想给第三方库封装一个易用的接口？
→ 使用 **外观模式**

**Q4**: 想在运行时动态切换实现？
→ 使用 **桥接模式**
